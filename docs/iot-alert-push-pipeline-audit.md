# IoT Alert Push Pipeline Audit

Audit date: 2026-05-25

Scope: `iot-metrics-collector-service`, `notification-service`, `common`, `config-server`, and mobile Android payload compatibility.

## 1. Executive Summary

Telemetry threshold alerts are mostly ready for Android push. The collector receives telemetry, stores readings, evaluates enabled `AlertRule`s, creates `AlertEvent`, publishes `AlertTriggeredEvent` to Kafka when `notifyWeb || notifyMobile`, and notification-service maps `notifyMobile=true` to FCM platforms `ANDROID` and `IOS`.

Disease detection alerts are only partially wired. The collector receives `image/meta`, creates `DeviceMediaEvent`, runs disease analysis, creates `AlertEvent` when disease is detected, and calls the same `AlertNotificationPublisher`. However, the disease `AlertEvent` has `alertRule = null`; `KafkaAlertNotificationPublisher` derives `notifyWeb` and `notifyMobile` only from `alertRule`, so both become `null`. Its required-field guard then skips Kafka publishing. Result: disease alerts can exist in the alert list, but they do not currently become `UserNotification` or FCM Android push.

Mobile Android routing is compatible with the current notification payload shape when the backend sends `type=IOT_ALERT` and `referenceId=<alertEventId>`. Telemetry alerts satisfy this after notification-service processing. Disease alerts would also route correctly after the backend publish gap is fixed because they already have an `AlertEvent` id.

## 2. Telemetry Threshold Alert Flow

Sequence observed:

```text
MQTT telemetry
-> MqttInboundMessageHandler.handleTelemetry(...)
-> TelemetryIngestServiceImpl.ingest(...)
-> save SensorReadingSeries
-> AggregateLatestReadingService.updateLatestReadings(...)
-> AlertEvaluationServiceImpl.evaluateReadings(...)
-> load enabled AlertRule by sensorTypeId
-> check min/max threshold
-> cooldown duplicate guard
-> save AlertEvent
-> KafkaAlertNotificationPublisher.publishAlertTriggered(...)
-> Kafka topic iot.alert.triggered
```

| Step | File/Class | Status | Notes |
| --- | --- | ---: | --- |
| Nhận MQTT telemetry | `MqttInboundMessageHandler` | Yes | `case "telemetry"` calls `handleTelemetry`, then `telemetryIngestService.ingest(...)`. |
| Lưu raw reading | `TelemetryIngestServiceImpl` | Yes | Saves `SensorReadingSeries` with `sensorReadingSeriesRepository.saveAll(readings)`. |
| Cập nhật latest | `TelemetryIngestServiceImpl` | Yes | Calls `aggregateLatestReadingService.updateLatestReadings(savedReadings)`. |
| Load enabled alert rules | `AlertEvaluationServiceImpl` | Yes | Uses `findAllByEnabledTrueAndSensorTypeId(...)`. |
| Check min/max threshold | `AlertEvaluationServiceImpl` | Yes | `minThreshold` maps to `THRESHOLD_LOW`; `maxThreshold` maps to `THRESHOLD_HIGH`. |
| Cooldown duplicate alert | `AlertEvaluationServiceImpl` | Yes | Checks active `OPEN`/`ACKNOWLEDGED` alert by rule/device/sensor since cooldown cutoff. |
| Save AlertEvent | `AlertEvaluationServiceImpl` | Yes | Saves with status `OPEN`, rule severity, owner, device, zone, sensor type, message. |
| Publish Kafka event | `AlertEvaluationServiceImpl` + `KafkaAlertNotificationPublisher` | Yes | Publish occurs only when `notifyWeb || notifyMobile`. |

Rule notification fields:

| Rule field | Default | Used in publish? | Notes |
| --- | ---: | ---: | --- |
| `notifyWeb` | `true` | Yes | Included in `AlertTriggeredEvent`; enables `IN_APP` and `WEB` FCM. |
| `notifyMobile` | `true` | Yes | Included in `AlertTriggeredEvent`; enables `ANDROID` and `IOS` FCM. |
| `enabled` | `true` | Yes | Only enabled rules are evaluated. |
| `cooldownMinutes` | nullable | Yes | `null` or `<=0` disables cooldown. |

Telemetry payload fields from `KafkaAlertNotificationPublisher`:

| Field | Telemetry status | Notes |
| --- | ---: | --- |
| `eventType` | Yes | Constant `ALERT_TRIGGERED`. |
| `alertEventId` | Yes | From saved `AlertEvent.id`. |
| `ownerUserId` | Yes | From alert owner, usually rule owner or device owner. |
| `deviceId` / `deviceUid` | Yes | From alert device. |
| `zoneId` / `farmPlotId` | Yes/Nullable | `zoneId` comes from alert zone; `farmPlotId` from device farm plot when present. |
| `sensorTypeCode` | Yes | From violating reading sensor type. |
| `alertType` | Yes | `THRESHOLD_LOW` or `THRESHOLD_HIGH`. |
| `severity` | Yes | From alert rule severity. |
| `triggerValue` / thresholds | Yes | From reading/rule. |
| `title` / `message` | Yes | Title is `<severity> <sensorCode> alert`; message is generated by evaluation service. |
| `notifyWeb` / `notifyMobile` | Yes | From `AlertRule`. |
| `referenceType` / `referenceId` | Yes | `referenceType=ALERT_EVENT`, `referenceId=alertEventId`. |
| `url` | Yes | `/dashboard/alerts?alertId=<alertEventId>`. |

## 3. Disease Detection Alert Flow

Sequence observed:

```text
Camera capture request / device upload
-> MQTT image/meta
-> MqttInboundMessageHandler.handleImageMeta(...)
-> DeviceMediaServiceImpl.handleImageMeta(...)
-> update DeviceMediaEvent status to UPLOADED
-> enqueue DeviceMediaAnalysisJob
-> DeviceMediaAnalysisServiceImpl.processJob(...)
-> file-service internal download URL
-> disease-detection-service predict endpoint
-> save DeviceMediaAnalysis
-> if disease detected, ImageDiseaseAlertServiceImpl.createDiseaseAlert(...)
-> save AlertEvent
-> KafkaAlertNotificationPublisher.publishAlertTriggered(...)
-> skipped because notifyWeb/notifyMobile are null
```

| Step | File/Class | Status | Notes |
| --- | --- | ---: | --- |
| Nhận MQTT image/meta | `MqttInboundMessageHandler` | Yes | `case "image/meta"` calls `deviceMediaService.handleImageMeta(...)`. |
| Tạo/cập nhật DeviceMediaEvent | `DeviceMediaServiceImpl` | Yes | Existing requested event is updated to `UPLOADED` or `FAILED` by `requestId`. |
| Lấy file/internal URL | `DeviceMediaAnalysisServiceImpl` + `FileServiceClient` | Yes | Uses internal download URL and rejects unusable presigned URLs. |
| Gọi disease detection | `DeviceMediaAnalysisServiceImpl` + `DiseaseDetectionFeignClient` | Yes | Sends image bytes as multipart to prediction endpoint. |
| Lưu analysis result | `DeviceMediaAnalysisServiceImpl` | Yes | Stores disease name/type, confidence, severity, notes, status. |
| Detect disease status | `DeviceMediaAnalysisServiceImpl` | Yes | `DISEASE_DETECTED` when class is not healthy and confidence >= configured threshold. |
| Tạo AlertEvent | `ImageDiseaseAlertServiceImpl` | Yes | Creates synthetic sensor type `CAMERA_DISEASE_DETECTION`, alert type `DISEASE_DETECTED`. |
| Publish Kafka event | `KafkaAlertNotificationPublisher` | No in practice | Called, but skipped by required-field guard because `notifyWeb` and `notifyMobile` are null. |

Disease alert details:

| Item | Status | Notes |
| --- | ---: | --- |
| Uses `AlertRule` | No | `ImageDiseaseAlertServiceImpl` sets `alertRule(null)`. |
| Has `notifyMobile/notifyWeb` control | No | No rule/config value is mapped into the alert event. |
| Has cooldown | No | Only avoids duplicate alert for the same `DeviceMediaAnalysis` because `analysis.getAlertEvent() == null`; repeated camera captures can create repeated alerts. |
| Has synthetic sensor type | Yes | `CAMERA_DISEASE_DETECTION` is created if missing. |
| Has AlertEvent id for routing | Yes | Analysis response includes linked `alertEventId`. |
| Publishes usable notification | No | Current publisher requires at least one notify flag to be true. |

Key blocker:

```text
ImageDiseaseAlertServiceImpl creates AlertEvent with alertRule = null.
KafkaAlertNotificationPublisher.toEvent(...) sets:
  notifyWeb = alertRule != null ? alertRule.notifyWeb : null
  notifyMobile = alertRule != null ? alertRule.notifyMobile : null
KafkaAlertNotificationPublisher.hasRequiredNotificationFields(...) requires:
  notifyWeb == true || notifyMobile == true
Therefore disease detection alerts are skipped before Kafka.
```

## 5. Notification-Service Delivery

Observed notification-service sequence:

```text
iot.alert.triggered
-> AlertTriggeredConsumer
-> validate eventId, ownerUserId, alertEventId
-> AlertReadyPublisher
-> iot.alert.ready
-> AlertReadyConsumer
-> PushNotificationServiceImpl.handleAlertTriggered(...)
-> create BatchedNotificationEvent type=IOT_ALERT
-> NotificationDeliveryServiceImpl.persist(...)
-> UserNotification
-> FcmDeliveryStrategy
-> Firebase Admin SDK
-> active push tokens filtered by ANDROID/IOS for notifyMobile
```

| Step | File/Class | Status | Notes |
| --- | --- | ---: | --- |
| Consume `iot.alert.triggered` | `AlertTriggeredConsumer` | Yes | Raw stage consumes the external topic. |
| Validate raw event | `AlertTriggeredConsumer` | Partial | Only checks `eventId`, `ownerUserId`, and `alertEventId`. |
| Publish `iot.alert.ready` | `AlertReadyPublisher` | Yes | Serializes event to JSON string and partitions by owner user. |
| Consume ready event | `AlertReadyConsumer` | Yes | Delegates to push service. |
| Validate delivery event | `PushNotificationServiceImpl` | Yes | Requires id, owner, device, sensor, severity, title, message, url. |
| Create UserNotification | `NotificationDeliveryServiceImpl` + persistence service | Yes | Uses `NotificationType.IOT_ALERT`. |
| Determine channels | `PushNotificationServiceImpl` | Yes | `notifyWeb=true` -> `IN_APP` + `FCM`; `notifyMobile=true` -> `FCM`. |
| Target Android/iOS | `PushNotificationServiceImpl` | Yes | `notifyMobile=true` adds `ANDROID` and `IOS`. |
| Build FCM data payload | `NotificationDeliveryServiceImpl` | Yes | Converts selected payload values to strings. |
| Send Firebase Admin FCM | `FcmDeliveryStrategy` | Yes | Uses `FirebaseMessaging.send(...)`. |
| Deactivate invalid token | `FcmDeliveryStrategy` | Yes | Deactivates on `UNREGISTERED` and `INVALID_ARGUMENT`. |

Token/platform model:

| Field | Meaning | Android compatibility |
| --- | --- | --- |
| `TokenDevice.userId` | Auth user id recipient | Required. |
| `TokenDevice.fcmToken` | Native FCM registration token | Required for Firebase Admin direct send. |
| `TokenDevice.platform` | Enum platform | Must be `ANDROID` for Android device. |
| `Platform` enum | `ANDROID`, `IOS`, `WEB` | No `EXPO` or `MOBILE`. |
| `RegisterPushTokenRequest.platform` | Client-provided enum | Android app must send `ANDROID`. |
| Gateway route | `/api/push-tokens/**` | Routed to notification-service. |

Compatibility conclusion:

Telemetry FCM payload is compatible with current mobile routing because `referenceId` equals `alertEventId`. Disease detection would also be compatible if the backend published the same `IOT_ALERT` payload. If a future disease notification used only `mediaEventId` or `analysisId`, current mobile routing would not open alert detail.

## 8. Required Fixes

Recommended fix plan:

### Phase P1 - Fix disease notification policy

Standardize disease detection alerts into the same notification contract:

```text
DeviceMediaAnalysis detects disease
-> ImageDiseaseAlertServiceImpl creates AlertEvent type=DISEASE_DETECTED
-> derive notifyWeb/notifyMobile from an explicit source
-> KafkaAlertNotificationPublisher publishes AlertTriggeredEvent
-> notification-service creates UserNotification + FCM
-> mobile tap opens AlertEventDetailScreen
```

Options for notification flags:

| Option | Notes |
| --- | --- |
| Add disease `AlertRule` support | Best for user control and consistency with telemetry thresholds. |
| Add device/camera disease notification config | Simpler if disease alerts are device-level rather than sensor-rule-level. |
| Add default policy in `ImageDiseaseAlertServiceImpl`/publisher | Fastest, but less controllable and can spam users. |

Recommended fields:

| Field | Recommended value |
| --- | --- |
| `alertType` | `DISEASE_DETECTED` |
| `severity` | Based on confidence: existing `MEDIUM/HIGH/CRITICAL` mapping is acceptable. |
| `message` | User-readable disease message, preferably localized later. |
| `deviceId`, `zoneId`, `farmPlotId` | From media event/device context. |
| `sensorTypeCode` | `CAMERA_DISEASE_DETECTION` is acceptable as a synthetic source. |
| `mediaEventId`, `analysisId` | Add to event/payload if the mobile detail should link to image/analysis. |
| `notifyWeb`, `notifyMobile` | Must be explicit booleans before publish. |
| `referenceId` | `alertEventId`. |

### Phase P2 - Add disease cooldown/deduplication

Add cooldown by `(device, zone, diseaseName)` or `(device, diseaseName)` so repeated camera captures do not generate many push notifications for the same condition.

### Phase P3 - Payload/mobile hardening

Add `alertEventId` fallback in mobile notification parsing and consider carrying disease-specific metadata (`mediaEventId`, `analysisId`, `diseaseName`, `confidence`) in FCM data. Mobile can still route by `referenceId`; extra data should only improve display.

## 9. Manual QA Checklist

### Telemetry threshold alert

1. Ensure Android app registers an active `ANDROID` FCM token in notification-service.
2. Ensure Firebase Admin is enabled in notification-service.
3. Ensure an enabled telemetry `AlertRule` has `notifyMobile=true`.
4. Send MQTT/HTTP telemetry crossing the configured threshold.
5. Check `AlertEvent` is created in iot database.
6. Check collector logs show publish to `iot.alert.triggered`.
7. Check notification-service raw stage consumes `iot.alert.triggered`.
8. Check ready stage consumes `iot.alert.ready`.
9. Check `UserNotification type=IOT_ALERT` is persisted.
10. Check `FcmDeliveryStrategy` sends to an `ANDROID` token.
11. Confirm Android receives notification.
12. Tap notification and confirm `/(main)/iot/alerts/<alertEventId>` opens.

### Disease detection alert

1. Capture or upload an image that returns disease above confidence threshold.
2. Check `DeviceMediaEvent` becomes `UPLOADED`.
3. Check `DeviceMediaAnalysis` becomes `DISEASE_DETECTED`.
4. Check linked `AlertEvent` is created.
5. With current code, expect Kafka publish to be skipped because notify flags are missing.
6. After Phase P1 fix, check `iot.alert.triggered` receives event.
7. Check notification-service creates `UserNotification type=IOT_ALERT`.
8. Check FCM sends to `ANDROID`.
9. Tap notification and confirm alert detail opens.

## 10. Open Questions

| Question | Why it matters |
| --- | --- |
| Should disease detection always create push alerts? | Controls noise and user expectations. |
| Should disease notification be controlled by `AlertRule` or device config? | Determines where `notifyMobile`, `notifyWeb`, and cooldown should live. |
| How should disease severity be determined? | Existing confidence mapping works technically, but may need agronomy rules. |
| What cooldown should disease alerts use? | Prevents repeated camera captures from spamming push. |
| Should mobile alert detail show media image/analysis? | Current route opens alert detail; richer disease context may need media fields. |
