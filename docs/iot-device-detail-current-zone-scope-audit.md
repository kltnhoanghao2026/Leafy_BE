# Device Detail Current Zone Scope Audit

## 1. Executive Summary

Business decision: `DeviceDetailPage` should default to current-zone scoped data. The page should show data generated by the device while it is assigned to its current `zoneId`, not the entire device history.

Current state:

- `DeviceDetailPage` receives current assignment from `GET /iot/devices/{deviceId}/detail`, specifically `DeviceDetailResponse.zoneId`.
- The page currently calls device-scoped endpoints:
  - `GET /iot/devices/{deviceId}/latest-readings`
  - `GET /iot/devices/{deviceId}/charts`
  - `GET /iot/devices/{deviceId}/media`
- These endpoints filter by `device_id` only and can return data from previous zones.
- The detail endpoint itself also embeds device-scoped `latestReadings`, `alertSummary`, and `latestMedia`.
- Backend data rows already have zone snapshots, but device-scoped endpoints do not expose `device_id + zone_id` filters yet.

Recommended first implementation: add optional `zoneId` query parameters to existing device data APIs. This preserves backward compatibility while allowing `DeviceDetailPage` to pass `device.zoneId` and render the current-zone view.

Do not change device-level settings:

- Device metadata/header remains device scoped.
- Device config remains device scoped.
- Camera schedules remain device scoped unless product later decides schedules must be reset on relocation.
- Capture/detect actions remain device scoped, but the media list after capture should be queried with current `zoneId`.

- `visibleReadings` uses `latestReadingsQuery.data` if available, otherwise falls back to `device.latestReadings`.

## 3. Backend Endpoint Gap

| Data | Current endpoint | Current filter | Can add `zoneId`? | Repository/service gap |
| --- | --- | --- | ---: | --- |
| Latest readings | `GET /iot/devices/{deviceId}/latest-readings` | `SensorLatestReading.device.id = :deviceId` | Yes | Add `findAllByDeviceIdAndZoneId(deviceId, zoneId)` |
| Device chart | `GET /iot/devices/{deviceId}/charts?sensorCode=&range=` | Aggregate `device.id = :deviceId` | Yes | Add aggregate methods filtering `device.id`, `zone.id`, `sensorType.id`, range |
| Device media | `GET /iot/devices/{deviceId}/media` | `DeviceMediaEvent.device.id = :deviceId` | Yes | Add `findTop20ByDeviceIdAndZoneIdOrderByRequestedAtDesc(deviceId, zoneId)` |
| Device detail embedded latest | `GET /iot/devices/{deviceId}/detail` | `findAllByDeviceId(deviceId)` | Optional | Either add `zoneId` param or make detail use current device zone for embedded data |
| Device detail embedded latest media | `GET /iot/devices/{deviceId}/detail` | `findTopByDeviceIdOrderByCapturedAtDesc(deviceId)` | Optional | Add `findTopByDeviceIdAndZoneIdOrderByCapturedAtDesc` |
| Device detail embedded alert summary | `GET /iot/devices/{deviceId}/detail` | `countByDeviceIdAndStatus` | Optional | Add `deviceId + zoneId + status` count methods if summary stays in response |
| Alert events list | `GET /iot/alert-events` | Supports `deviceId`; supports `zoneId` separately | Already possible | FE can send both `deviceId` and `zoneId` if service composes filters with AND |

Backend facts:

- `TelemetryQueryController` device methods do not accept `zoneId`.
- `TelemetryQueryService` has device methods and zone methods, but no combined device+zone method.
- Aggregate repositories already have device-only and zone-only query methods. They do not have list methods for `deviceId + zoneId + sensorType + time range`.
- Aggregate repositories do have single-row upsert lookup methods with `deviceId + zoneId`, but those are not list queries for charts.
- `DeviceMediaService.listDeviceMedia` accepts only `deviceId`.
- `DeviceMediaEventRepository` has device-only and zone-only latest methods, but no device+zone list method.

## 4. Recommended API Contract

Use optional query parameters on existing endpoints:

```http
GET /iot/devices/{deviceId}/latest-readings?zoneId={zoneId}
GET /iot/devices/{deviceId}/charts?sensorCode=...&range=...&zoneId={zoneId}
GET /iot/devices/{deviceId}/media?zoneId={zoneId}
```

Behavior:

- If `zoneId` is omitted, preserve current device-history behavior.
- If `zoneId` is provided, filter by both `deviceId` and stored event/aggregate/latest `zone_id`.
- `DeviceDetailPage` should pass `device.zoneId`.
- If `device.zoneId` is null, current-zone scoped data queries should be disabled or return empty state.
- This keeps a path for a future "Device history" tab by calling the same endpoints without `zoneId`.

Optional detail endpoint change:

```http
GET /iot/devices/{deviceId}/detail?zoneId={zoneId}
```

Two viable approaches:

1. Keep detail as metadata-only for current page usage and rely on the separate scoped latest/media/chart calls.
2. Add `zoneId` to detail too so embedded `latestReadings`, `latestMedia`, and `alertSummary` are scoped consistently.

Recommendation: implement option 2 or remove/ignore embedded data fallback in FE current-zone mode. Leaving device-scoped embedded fallback is a subtle bug source.

Access control:

- Continue `requireOwnedDevice(deviceId, currentUserId)`.
- Also verify the requested `zoneId` is either the device's current zone or an owned zone.
- For the current business decision, prefer requiring `zoneId == device.zone.id` in current-zone DeviceDetail mode. This prevents clients from querying arbitrary old zone data through the device endpoint.
- If a future device-history tab needs old assignment data, that should be a separate product decision and may require assignment history.

## 5. Frontend Hook/Query Key Changes

Current hooks:

```ts
useDeviceLatestReadings(deviceId)
useDeviceChart(deviceId, sensorCode, range)
useDeviceMedia(deviceId)
```

Recommended hook signatures:

```ts
useDeviceLatestReadings(deviceId, zoneId, enabled)
useDeviceChart(deviceId, sensorCode, range, zoneId, enabled)
useDeviceMedia(deviceId, zoneId, enabled, refetchInterval)
```

Recommended query keys:

```ts
deviceKeys.latestReadings(deviceId, zoneId)
deviceKeys.chart(deviceId, sensorCode, range, zoneId)
deviceKeys.media(deviceId, zoneId)
```

`zoneId` must be part of the key. Otherwise React Query can reuse old device-history cache for current-zone scoped data.

Recommended API wrapper changes:

```ts
collectorApi.getDeviceLatestReadings(deviceId, zoneId)
collectorApi.getDeviceChart(deviceId, sensorCode, range, zoneId)
collectorApi.getDeviceMedia(deviceId, zoneId)
```

Expected `DeviceDetailPage` changes:

- Derive `currentZoneId = device?.zoneId ?? ""`.
- Enable latest/media/chart current-zone queries only when `deviceId` and `currentZoneId` are present.
- Pass `currentZoneId` into `DeviceSensorCard`, because chart fetching currently happens inside that component.
- Pass `zoneId` into `useAlertEvents({ deviceId, zoneId })` if chart markers/threshold analytics must match current zone.
- Avoid fallback from scoped latest query to `device.latestReadings` unless detail endpoint is also scoped.
- Show a relocation/current-zone empty notice when there is a current zone but no scoped readings/media/chart data.

Do not change these hooks to zone scope:

- `useDeviceDetail(deviceId)` for metadata.
- `useDeviceConfig(deviceId)`.
- `useDeviceSchedulesQuery(deviceUid)`.
- Config update/push mutations.
- Capture mutation.

Mutation invalidation:

- `useCaptureDeviceImage(deviceId)` should invalidate `deviceKeys.media(deviceId, currentZoneId)` after capture when FE knows current zone.
- Existing invalidation of `deviceKeys.media(deviceId)` should be broadened or changed to invalidate all media keys under the device/all prefix.
- `useUpdateDeviceMutation` should invalidate old and new zone-scoped keys where possible. If old zone is not known in mutation variables, invalidating `deviceKeys.all()` and `metricsKeys.all()` remains acceptable.

## 6. Expected Behavior After Relocation

When a device moves from zone A to zone B:

- Device header shows current farm/zone B.
- Latest readings section queries `deviceId + zoneB`.
- If no telemetry has arrived in zone B, latest readings show empty state.
- Charts query `deviceId + zoneB`; zone A aggregate points are excluded.
- Media/diagnosis query `deviceId + zoneB`; zone A images and disease analysis are excluded.
- Alert markers query `deviceId + zoneB`; zone A alert markers are excluded from current-zone charts.
- Config and camera schedules remain visible because they are device-level settings.
- Manual capture still works; backend snapshots the current device zone on the new media event.

Recommended empty/notice copy:

```text
Thiết bị đang được gắn vào khu vực này. Dữ liệu mới sẽ xuất hiện sau khi thiết bị gửi mẫu hoặc chụp ảnh mới trong khu vực hiện tại.
```

This notice should appear when:

- the device has a current `zoneId`,
- current-zone scoped latest/media/chart data is empty,
- and the device is claimed/active enough that data is expected.

If `device.zoneId` is null:

- show unassigned device state,
- keep config/schedules visible if allowed,
- disable current-zone latest/chart/media sections or show "Thiết bị chưa được gắn khu vực".

## 7. Tests Needed

Backend tests:

- `GET /iot/devices/{deviceId}/latest-readings?zoneId=zoneB` returns latest rows for device+zoneB and excludes device+zoneA.
- `GET /iot/devices/{deviceId}/latest-readings` without `zoneId` preserves current device-only behavior.
- `GET /iot/devices/{deviceId}/charts?...&zoneId=zoneB` returns aggregate points for device+zoneB and excludes zoneA.
- `GET /iot/devices/{deviceId}/charts` without `zoneId` preserves current device-only behavior.
- `GET /iot/devices/{deviceId}/media?zoneId=zoneB` returns media events for device+zoneB and excludes zoneA.
- `GET /iot/devices/{deviceId}/media` without `zoneId` preserves current device-only behavior.
- If detail endpoint adds `zoneId`, `GET /iot/devices/{deviceId}/detail?zoneId=zoneB` scopes embedded latest/media/alert summary.
- Access-control test: user cannot request a zone not owned or not equal to current device zone if strict current-zone rule is chosen.

Frontend tests:

- `DeviceDetailPage` waits for `device.zoneId` and calls latest/media/chart endpoints with `zoneId`.
- Query keys include `zoneId`; changing device zone does not reuse old cache.
- If current-zone latest readings are empty, current-zone empty notice appears.
- Device config section still renders from device config endpoint.
- Camera schedule section still uses device UID scope.
- `DeviceSensorCard` passes `zoneId` to chart hook.
- Alert query includes both `deviceId` and `zoneId` if markers remain on charts.
- Media panel does not render media with a different `zoneId` in mocked responses.

## 8. Implementation Phases

### Phase 1 - Backend optional `zoneId` filters

- Add optional `zoneId` to:
  - `TelemetryQueryController.getLatestReadingsByDevice`
  - `TelemetryQueryController.getDeviceSensorChart`
  - `DeviceController.getDeviceMedia`
- Add service methods or overloads for `deviceId + zoneId`.
- Add repository list methods:
  - `SensorLatestReadingRepository.findAllByDeviceIdAndZoneId`
  - `SensorReadingAgg5mRepository.findAllByDeviceIdAndZoneIdAndSensorTypeId...`
  - `SensorReadingAgg1hRepository.findAllByDeviceIdAndZoneIdAndSensorTypeId...`
  - `SensorReadingAgg1dRepository.findAllByDeviceIdAndZoneIdAndSensorTypeId...`
  - `DeviceMediaEventRepository.findTop20ByDeviceIdAndZoneIdOrderByRequestedAtDesc`
- Keep no-`zoneId` behavior unchanged.

### Phase 2 - Frontend current-zone DeviceDetailPage

- Update `collectorApi`, hooks, and query keys to accept `zoneId`.
- Pass `device.zoneId` into latest/chart/media hooks.
- Pass `device.zoneId` into `useAlertEvents` for chart markers.
- Prevent fallback to device-history embedded `latestReadings` unless detail endpoint is scoped.
- Add current-zone empty/relocation notice.

### Phase 3 - Detail endpoint consistency

- Either add `zoneId` to `GET /iot/devices/{deviceId}/detail`, or stop using embedded latest/media/alert summary for current-zone page data.
- If adding `zoneId`, scope:
  - `latestReadings`
  - `latestMedia`
  - `alertSummary`
- Keep metadata/config device-scoped.

### Phase 4 - Zone page media/diagnosis

- Fix `ZoneDetailMetricsPage` separately so it no longer calls `useDeviceMedia(zoneDeviceId)` for zone history.
- Prefer zone media endpoint or a new endpoint accepting `zoneId`.

### Phase 5 - Latest relocation policy

- Decide whether to reset, stale-mark, or retain latest readings when device changes zone.
- This is separate from query scoping. Query scoping prevents wrong display, but backend still lacks explicit relocation semantics.
